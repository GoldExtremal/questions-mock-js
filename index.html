Контекст и замыкания

1. Что происходит в движке JavaScript после запуска файла?

После запуска JavaScript-файла движок сначала парсит код, проверяет его на синтаксические ошибки, строит абстрактное синтаксическое дерево (структура, с которой движку удобно работать), создаёт глобальный контекст выполнения и подготавливает окружение, после этого начинается выполнение программы.

Что такое глобальный контекст выполнения?

Глобальный контекст выполнения — это первый контекст выполнения, который создаётся при запуске программы. Он представляет собой среду, в которой исполняется весь код верхнего уровня, и содержит глобальный объект, this, области видимости, переменные и функции.

Какие фазы проходит создание контекста?
  • Фаза создания — движок выделяет память, создаёт глобальный объект и this, формирует области видимости и регистрирует все объявления переменных и функций.
  • Фаза выполнения — после подготовки окружения движок начинает выполнять код сверху вниз.
Что хранится в памяти на старте?

Во время фазы создания в памяти создаются все объявления:

  • var — создаётся и сразу получает значение undefined;
  • let и const — создаются, но находятся в состоянии uninitialized (Temporal Dead Zone);
  • function declaration — сразу сохраняется как готовый объект функции и доступна до строки объявления.

Именно благодаря этой предварительной подготовке памяти возникает эффект hoisting.

Что происходит на этапе выполнения?

Во время фазы выполнения движок выполняет код сверху вниз: присваивает значения переменным, вычисляет выражения и вызывает функции. При каждом вызове функции создаётся новый контекст выполнения, который помещается в стек вызовов. После завершения функции её контекст удаляется из стека.

2. Что происходит при вызове функции?

При вызове функции движок создаёт для неё отдельную среду выполнения, чтобы изолировать её локальные данные и выполнить код независимо от других функций.

Создаётся ли новый контекст выполнения?

Да. При каждом вызове функции создаётся новый контекст выполнения. Он содержит локальные переменные, параметры функции, значение this, а также ссылки на внешнее лексическое окружение.

Что происходит со стеком вызовов?

Новый контекст выполнения помещается на вершину стека вызовов. Пока он находится в стеке, движок выполняет код этой функции. Если внутри вызывается другая функция, её контекст также помещается в стек поверх предыдущего.

Когда контекст исчезает?

После завершения выполнения функции или после выполнения оператора return её контекст выполнения удаляется из стека вызовов, и управление возвращается в предыдущий контекст.

Почему локальные переменные обычно удаляются?

После удаления контекста выполнения локальные переменные становятся недоступны, так как ссылки на них больше не существуют. Если на эти данные нет других ссылок, они считаются недостижимыми и впоследствии освобождаются сборщиком мусора.

Важно: если локальные переменные были захвачены замыканием, они не удаляются после завершения функции, так как на них продолжают существовать ссылки.

3. Что такое лексическое окружение?

Лексическое окружение — это внутренняя структура данных, в которой движок JavaScript хранит переменные, функции и информацию о внешней области видимости. Оно создаётся для каждого контекста выполнения.

Что хранится внутри?

Лексическое окружение состоит из двух частей:

  • Запись окружения (Environment Record) — хранит объявления переменных, функций и параметров текущей области видимости.
  • Ссылка на внешнее лексическое окружение, в котором была создана текущая функция.
Что такое запись окружения (Environment Record)?

Запись окружения (Environment Record) — это часть лексического окружения, которая хранит все идентификаторы текущей области видимости: переменные (var, let, const), параметры функции и объявления функций, а также их текущие значения.

Как оно связано с цепочкой областей видимости (Scope Chain)?

Цепочка областей видимости (Scope Chain) строится благодаря ссылкам на внешние лексические окружения. Если переменная не найдена в текущем Environment Record, движок переходит во внешнее лексическое окружение, затем в следующее и так далее, пока не найдёт переменную или не дойдёт до глобальной области видимости. Именно этот механизм обеспечивает работу лексической области видимости и замыканий.

4. Как происходит поиск переменной?

Поиск переменной происходит по цепочке областей видимости (Scope Chain). Движок последовательно проверяет текущую область видимости, а затем при необходимости переходит во внешние лексические окружения.

Сначала переменная ищется в Environment Record текущего лексического окружения. Если она не найдена, движок переходит по ссылке во внешнюю область видимости.

Поиск прекращается в двух случаях:

  1. Если переменная найдена - её значение сразу возвращается;
  2. Если достигнуто глобальное лексическое окружение и переменная не найдена, дальнейший поиск прекращается.

Если после проверки всей Scope Chain переменная не найдена, при попытке обратиться к ней будет выброшено ReferenceError.

Важно: это касается только необъявленных переменных. Если переменная существует, но имеет значение undefined, ошибки не будет — движок вернёт undefined.

5. Что такое замыкание?

Замыкание — это функция, которая сохраняет доступ к переменным своего внешнего лексического окружения даже после завершения выполнения внешней функции.

Почему переменная не удаляется?

Обычно после завершения функции её контекст выполнения удаляется. Однако если внутренняя функция продолжает использовать переменные внешней функции, движок сохраняет их в памяти, так как они всё ещё нужны замыканию.

Где хранится ссылка?

Ссылка хранится в лексическом окружении внутренней функции, а именно в её Outer Lexical Environment Reference, который указывает на лексическое окружение внешней функции. Благодаря этой ссылке замыкание сохраняет доступ к внешним переменным.

Когда память всё же освобождается?

Память освобождается, когда на замыкание больше не остаётся ссылок. После этого лексическое окружение становится недостижимым, и Garbage Collector может удалить его из памяти.

Может ли замыкание привести к утечке памяти?

Да. Само по себе замыкание не является утечкой памяти, но если оно продолжает удерживать ссылки на ненужные объекты или долго живёт без необходимости, эти данные не смогут быть удалены сборщиком мусора, что приведёт к лишнему потреблению памяти.

6. Что такое this и от чего он зависит?

this — это значение, которое функция получает при вызове и через которое обычно обращается к текущему объекту или контексту выполнения.

У обычных функций this определяется не местом объявления, а способом вызова. Одна и та же функция может получить разный this, если вызвать её как метод объекта, обычную функцию, через call, apply, bind или через new.

Когда определяется this?

this определяется в момент вызова функции. До вызова у обычной функции нет окончательного значения this, потому что ещё неизвестно, каким способом её запустят.

От чего зависит this?

Он зависит от формы вызова: при вызове как метода this обычно указывает на объект перед точкой, при обычном вызове в строгом режиме будет undefined, через call, apply и bind задаётся явно, а при вызове через new указывает на создаваемый объект.

Почему this нельзя определить по месту объявления функции?

Потому что обычная функция не хранит постоянную привязку к объекту, рядом с которым она была объявлена. Важен не сам текст функции, а то, как именно ссылка на эту функцию была вызвана.

Важно: стрелочные функции работают иначе. У них нет собственного this, они берут его из внешнего лексического окружения.

Проверим, что происходит, когда метод сохраняют в переменную и вызывают отдельно от объекта.

const obj = {
    value: 10,
    show() {
        console.log(this.value);
    }
};

const fn = obj.show;
fn();

7. Что выведет этот код и почему?

В строгом режиме этот код выбросит ошибку TypeError: при вызове fn() значение this будет undefined, поэтому обращение к this.value невозможно.

В нестрогом режиме this будет указывать на глобальный объект. Если на нём нет свойства value, в консоль выведется undefined. В любом случае результат не будет равен 10, потому что функция уже не вызывается как метод объекта obj.

Почему получился именно такой результат?

Потому что в строке const fn = obj.show; в переменную попадает только ссылка на функцию. Связь с объектом obj при этом не сохраняется. Когда дальше вызывается fn(), это уже обычный вызов функции, а не вызов метода объекта.

Как сохранить правильный this?

Нужно сохранить способ вызова, при котором объект остаётся контекстом. Самый простой вариант — вызвать метод напрямую: obj.show(). Если функцию нужно передать отдельно, контекст надо привязать заранее через bind.

Какие есть способы исправить вызов?

Можно вызвать метод напрямую: obj.show(). Можно явно передать контекст при вызове: fn.call(obj) или fn.apply(obj). Можно создать новую привязанную функцию: const fn = obj.show.bind(obj);. Ещё один вариант — использовать обёртку: () => obj.show().

8. Как работают call, apply и bind?

call, apply и bind нужны, чтобы явно задать this для обычной функции.

call и apply сразу вызывают функцию. Разница между ними только в передаче аргументов: call принимает их по одному, а apply принимает массив или массивоподобное значение. bind не вызывает функцию сразу, а возвращает новую функцию с заранее привязанным this и, при необходимости, частью аргументов.

Чем call отличается от apply?

call передаёт аргументы через запятую: fn.call(obj, a, b). apply передаёт аргументы одним списком: fn.apply(obj, [a, b]).

Что возвращает bind?

bind возвращает новую функцию. Она запоминает переданный this и начальные аргументы, но выполнится только тогда, когда её потом вызовут.

Можно ли изменить this у уже привязанной функции?

Обычно нет. Если функция уже создана через bind, повторный bind, call или apply не поменяют её исходный привязанный this.

9. Чем стрелочные функции отличаются от обычных?

Стрелочная функция не создаёт собственные this, arguments и prototype. Главное отличие в том, что this у неё берётся из внешнего лексического окружения.

Поэтому стрелки удобны внутри колбэков и вложенных функций, когда нужно сохранить внешний this. Но они плохо подходят для методов объекта, если метод должен получать this от способа вызова.

Откуда стрелочная функция берёт this?

Она берёт this из окружающего кода. В примере стрелка создаётся внутри метода foo, поэтому использует тот this, который был у foo во время вызова.

Почему call, apply и bind на неё не влияют?

Потому что у стрелочной функции нет собственного this, который можно было бы заменить. call, apply и bind могут передать значение, но стрелка его не использует.

Почему стрелочную функцию нельзя вызвать через new?

Стрелочная функция не является конструктором и не имеет собственного prototype. Поэтому она не может создавать экземпляры через new.

10. Как работает new?

new запускает функцию-конструктор в специальном режиме создания объекта. Он подготавливает новый объект, связывает его с прототипом конструктора, вызывает конструктор с этим объектом в качестве this и возвращает результат.

Какие шаги выполняет new?
  • Создаёт новый пустой объект.
  • Связывает внутренний прототип объекта с Constructor.prototype.
  • Вызывает конструктор так, что this внутри него указывает на новый объект.
  • Возвращает созданный объект, если конструктор явно не вернул другой объект.
Почему после этого работает instanceof?

instanceof проверяет, есть ли User.prototype в цепочке прототипов объекта. После вызова new User() эта связь появляется, поэтому user instanceof User возвращает true.

Как prototype участвует в создании объекта?

prototype берётся у функции-конструктора. Новый объект получает ссылку на User.prototype, и благодаря этому может находить методы и свойства в цепочке прототипов.

Важно: если конструктор явно возвращает объект, результатом new будет этот объект. Если возвращается примитив, он игнорируется.

Пока занят Call Stack, новый код не запускается. Когда стек пуст, среда берёт следующую работу из очередей.

Сам JavaScript выполняет код и управляет стеком вызовов. Таймеры, события, сетевые операции и петлю событий даёт среда.

Что добавляет среда выполнения?

Браузер или Node.js добавляют API, очереди задач и правила возврата колбэков в выполнение.

В каком порядке выполняются Call Stack, microtasks и macrotasks?

Сначала выполняется синхронный код. Затем очищаются microtasks, например Promise.then. Потом берётся следующая macrotask, например setTimeout.

Важно: Event Loop — не часть синтаксиса языка, а механизм среды вокруг движка.

12. Как работает Call Stack?

Call Stack — стек активных вызовов функций. Последний вызов лежит сверху и завершается первым.

Какие функции попадают в стек?

В стек попадает каждый активный вызов: сначала глобальный контекст, затем foo(), bar() и baz().

Как меняется стек по шагам?

При вызове контекст добавляется наверх, при завершении удаляется. Стек растёт как foo → bar → baz и разбирается обратно.

Что произойдёт при слишком глубокой рекурсии?

При слишком глубокой рекурсии стек переполняется, и движок выбрасывает RangeError: Maximum call stack size exceeded.

13. В чём разница между microtasks и macrotasks?

Microtasks выполняются сразу после текущего синхронного кода, до перехода к следующей задаче. Macrotasks выполняются отдельными шагами event loop: например, таймеры, события и сетевые колбэки.

Что выведет этот код?

Сначала выполняется синхронный код: start и end. Затем микрозадача промиса выводит promise, и только потом задача таймера выводит timeout.

Почему Promise срабатывает раньше setTimeout?

После очистки стека сначала выполняются microtasks. Поэтому Promise.then идёт раньше setTimeout.

Когда очищается очередь микрозадач?

Она очищается после текущего синхронного кода и до следующей macrotask. Новые микрозадачи тоже выполняются в этом же цикле.

Может ли очередь микрозадач надолго заблокировать страницу?

Да. Бесконечная цепочка микрозадач не даст браузеру перейти к событиям и отрисовке.

14. Как работают async / await?

async / await — синтаксис поверх Promise, который делает асинхронный код похожим на синхронный.

Что возвращает async-функция?

async-функция всегда возвращает Promise: обычное значение станет fulfilled, ошибка — rejected.

Что происходит на await?

await приостанавливает только текущую async-функцию. Её продолжение попадёт в очередь микрозадач после завершения промиса.

Почему основной поток не блокируется?

await отдаёт управление наружу, поэтому синхронный код после вызова run() продолжает выполняться.

15. Что такое Promise и как он работает?

Promise — объект для результата асинхронной операции: успеха или ошибки.

Что выведет этот код?

Код выведет ok, затем done. reject уже не сработает, потому что первым был resolve("ok").

Почему состояние Promise меняется только один раз?

Промис переходит из pending только один раз: в fulfilled или rejected. После этого состояние не меняется.

Зачем нужен finally?

finally нужен для общего завершающего действия: скрыть loader, закрыть соединение, освободить ресурс.

all ждёт все и падает при первом reject. race берёт первый завершённый. any берёт первый успешный. allSettled ждёт все статусы.

Практика по блоку

Нужно вернуть Promise, который собирает результаты по исходным индексам.

Решение задачи
function promiseAll(promises) {
    return new Promise((resolve, reject) => {
        const results = [];
        let completed = 0;

        if (promises.length === 0) {
            resolve(results);
            return;
        }

        promises.forEach((promise, index) => {
            Promise.resolve(promise)
                .then((value) => {
                    results[index] = value;
                    completed += 1;

                    if (completed === promises.length) {
                        resolve(results);
                    }
                })
                .catch(reject);
        });
    });
}

const first = new Promise((resolve) => {
    setTimeout(() => resolve("first"), 300);
});

const second = new Promise((resolve) => {
    setTimeout(() => resolve("second"), 100);
});

const third = new Promise((resolve) => {
    setTimeout(() => resolve("third"), 200);
});

promiseAll([first, second, third]).then(console.log);
// ["first", "second", "third"]
Как сохранить порядок результатов?

Записывай результат по индексу исходного массива, а не через push. Так порядок сохранится независимо от скорости промисов.

Что делать при первом reject?

При первом отклонении сразу вызывай reject итогового промиса.

16. Что такое прототип и зачем он нужен?

Прототип — это объект, к которому другой объект обращается при поиске свойств и методов. Он нужен, чтобы объекты могли переиспользовать общее поведение без копирования методов в каждый экземпляр.

В примере метод sayHi лежит в User.prototype, поэтому все объекты, созданные через new User(), могут найти этот метод через цепочку прототипов.

Чем prototype отличается от __proto__?

prototype принадлежит функции-конструктору и задаёт прототип будущих экземпляров. __proto__ указывает на реальный прототип уже созданного объекта.

Что такое [[Prototype]]?

[[Prototype]] — внутренняя ссылка объекта на его прототип. Через неё движок поднимается выше по цепочке при поиске свойства.

Почему методы часто кладут в prototype?

Так экземпляры используют одну общую функцию вместо отдельных копий метода в каждом объекте.

17. Как происходит поиск свойства в цепочке прототипов?

Свойство ищется сначала на объекте, затем выше по цепочке [[Prototype]].

Где сначала ищется свойство?

Сначала проверяются собственные свойства объекта. В примере у user есть name, но нет role.

Когда поиск переходит в прототип?

Если свойства нет на объекте, движок проверяет прототип. Поэтому user.role находится в base.

Что будет, если свойство нигде не найдено?

Если цепочка дошла до null, чтение вернёт undefined.

18. Зачем нужен Object.create()?

Object.create() создаёт объект с явно заданным прототипом.

Что делает Object.create()?

Object.create(proto) создаёт объект с [[Prototype]], равным proto. null создаёт объект без прототипа.

Почему объект без Object.prototype иногда полезен?

Он удобен как словарь: нет унаследованных ключей вроде toString или constructor.

Какие ограничения есть у такого объекта?

У него нет методов из Object.prototype, поэтому вместо obj.hasOwnProperty лучше использовать статические проверки.

19. Как работает instanceof?

instanceof проверяет, есть ли объект из Constructor.prototype в цепочке прототипов проверяемого объекта.

Что проверяет instanceof?

Он не проверяет имя класса или факт вызова конкретного конструктора. Проверка сводится к обходу цепочки [[Prototype]]: если в ней найден User.prototype, результат будет true.

Как prototype участвует в проверке?

При new User() новый объект получает ссылку на текущий User.prototype. Поэтому user instanceof User возвращает true, пока этот объект действительно есть в цепочке прототипов.

Когда результат может быть неожиданным?

Если заменить User.prototype после создания экземпляра, старый объект не начнёт ссылаться на новый prototype автоматически. Также результат может измениться при ручной подмене цепочки через Object.setPrototypeOf() или при работе с объектами из другого JavaScript-контекста.

function User() {}

const user = new User();

User.prototype = {};

console.log(user instanceof User); // false

20. Что такое class в JavaScript?

class — удобный синтаксис поверх прототипной модели JavaScript.

Почему class называют синтаксическим сахаром?

Потому что class не меняет модель объектов, а только упрощает запись конструктора, методов и наследования.

extends связывает прототипы классов, поэтому Dog находит методы из Animal.prototype.

21. Что такое DOM?

DOM — объектная модель документа: дерево объектов, созданное браузером из HTML.

Что такое DOM-дерево?

DOM-дерево состоит из узлов: документа, элементов, текста и комментариев. Связи повторяют вложенность HTML.

Где хранится DOM?

DOM хранится в памяти браузера. Это не HTML-файл, а текущее представление страницы.

Кто создаёт DOM?

DOM создаёт браузер при парсинге HTML. JavaScript потом меняет его через DOM API.

22. Как работает всплытие события?

Когда в DOM происходит событие, оно доходит до целевого элемента, а затем может пройти обратно вверх по дереву. Эта обратная фаза называется всплытием события.

Например, если кликнуть по вложенной кнопке, событие сначала попадёт в обработчик самой кнопки, затем поднимется к её родителю, потом к следующим родителям выше по DOM-дереву.

В каком порядке сработают обработчики?

В текущем примере при клике по кнопке сначала выполнится обработчик кнопки и выведет button. После этого событие всплывёт до document.body, и второй обработчик выведет body.

Итоговый порядок вывода: button, затем body.

Что такое захват события?

Захват события — это фаза до достижения целевого элемента: событие идёт сверху вниз, от документа к цели. Её включают через { capture: true }.

В примере обработчик на document.body сработает до обработчика кнопки, потому что он выполняется на фазе захвата.

document.body.addEventListener("click", () => {
    console.log("body capture");
}, { capture: true });

button.addEventListener("click", () => {
    console.log("button");
});

// body capture
// button

На практике захват используют реже, чем всплытие. Он полезен, когда родительскому элементу нужно увидеть событие раньше дочерних обработчиков: например, для раннего логирования, глобальной проверки клика, перехвата события до стороннего виджета или работы с событиями, которые плохо подходят для обычного всплытия.

Для большинства UI-задач чаще выбирают всплытие и делегирование событий, потому что такая модель проще читается и лучше подходит для динамических элементов.

Как остановить всплытие?

event.stopPropagation() останавливает дальнейшее движение события по дереву. Если вызвать его в обработчике кнопки, обработчик на document.body уже не сработает.

Это полезно, когда вложенный элемент должен обработать событие сам и не запускать логику родителя. Например, клик по кнопке внутри карточки не должен открывать всю карточку, клик внутри модального окна не должен закрывать оверлей, а действие внутри dropdown не должно срабатывать как внешний клик.

Использовать это стоит аккуратно: остановка всплытия может сломать делегирование событий и общие обработчики выше по дереву.

Разберём, как один обработчик на родителе может заменить множество обработчиков на дочерних элементах.

list.addEventListener("click", (event) => {
    const item = event.target.closest("li");

    if (!item) return;

    console.log(item.dataset.id);
});

23. Как работает делегирование событий?

Делегирование — это один обработчик на родителе вместо обработчиков на каждом дочернем элементе.

Как понять, по какому элементу кликнули?

Берём event.target и ищем нужный элемент через closest(). Если его нет, выходим.

Почему это удобнее и часто быстрее?

Один обработчик проще поддерживать, и он работает даже для элементов, добавленных позже.

Какие ограничения есть у делегирования?

Делегирование работает через всплытие, поэтому оно плохо подходит для событий, которые не всплывают или ведут себя нестандартно. Например, focus и blur не всплывают как обычный click; для них чаще используют focusin и focusout.

Нужно аккуратно проверять границы контейнера. event.target.closest("li") может найти элемент выше или вне ожидаемой области, если не проверить, что найденный элемент действительно принадлежит текущему списку.

Ещё одно ограничение — вложенные интерактивные элементы. Например, клик по кнопке внутри карточки может одновременно запустить обработчик кнопки и обработчик карточки, если не разделить сценарии или не остановить всплытие там, где это оправдано.

24. Когда нужен MutationObserver?

MutationObserver нужен, когда нужно отследить изменения DOM, которые не являются обычными пользовательскими событиями: добавление или удаление узлов, изменение атрибутов, текста или вложенного дерева.

Это не замена click, input или другим событиям пользователя. Observer полезен именно тогда, когда важно узнать, что DOM уже изменился.

В каких случаях MutationObserver полезен?

Он помогает, когда DOM меняет не твой прямой обработчик, а сторонний виджет, редактор, расширение, асинхронно загруженный компонент или код, к которому нет удобной точки подключения.

Например, можно отследить появление элемента в контейнере, изменение атрибута, вставку новых сообщений в чат или перерисовку части страницы сторонней библиотекой.

Что приходит в callback?

В callback приходит массив MutationRecord. В каждой записи есть тип изменения, например childList, attributes или characterData, а также добавленные и удалённые узлы, имя изменённого атрибута и ссылка на изменённый target.

Callback MutationObserver выполняется как микрозадача: браузер не вызывает его синхронно на каждое изменение, а собирает изменения и вызывает callback после текущего синхронного кода, до перехода к следующей задаче.

Почему это не обычное событие?

Обычное событие описывает действие или сигнал: клик, ввод, загрузку, фокус. MutationObserver сообщает не о причине, а о результате — DOM изменился.

Из-за этого observer лучше использовать точечно. Если наблюдать слишком большой subtree или делать тяжёлую работу в callback, можно получить лишнюю нагрузку на страницу.

25. Что такое Reflow / Repaint и почему это важно?

Reflow и Repaint — этапы обновления страницы после изменений DOM, CSS или размеров окна. Они важны, потому что могут быть дорогими: браузеру приходится пересчитывать layout и заново рисовать элементы.

Чем reflow отличается от repaint?

Reflow — это пересчёт геометрии страницы: размеров, координат, положения элементов и влияния одних элементов на другие. Его вызывают изменения вроде ширины, высоты, шрифта, добавления узла или изменения классов, влияющих на layout.

Repaint — это перерисовка внешнего вида без обязательного пересчёта геометрии. Например, изменение цвета текста или фона обычно требует repaint, но не всегда требует reflow.

Почему чтение offsetHeight внутри цикла может замедлить страницу?

Если сначала изменить DOM или классы, а затем сразу прочитать layout-свойство вроде offsetHeight, offsetWidth или getBoundingClientRect(), браузер может принудительно пересчитать layout, чтобы вернуть актуальное значение.

В цикле это особенно плохо: каждая итерация может чередовать запись и чтение, вызывая серию forced reflow. Такой паттерн часто называют layout thrashing.

Как уменьшить лишние пересчёты layout?

Главное правило — группировать операции: сначала сделать все чтения размеров, потом все записи в DOM. Не стоит в одном цикле постоянно менять класс и сразу читать offsetHeight.

Также помогают CSS-классы вместо множества inline-изменений, пакетные DOM-операции, requestAnimationFrame для визуальных обновлений и уменьшение области изменений, чтобы браузеру не приходилось пересчитывать большую часть страницы.

Реализовать Promise.allSettled.

Решение
function promiseAllSettled(promises) {
    return new Promise((resolve) => {
        const results = [];
        let completed = 0;

        if (promises.length === 0) {
            resolve(results);
            return;
        }

        promises.forEach((promise, index) => {
            Promise.resolve(promise)
                .then((value) => {
                    results[index] = { status: "fulfilled", value };
                })
                .catch((reason) => {
                    results[index] = { status: "rejected", reason };
                })
                .finally(() => {
                    completed += 1;

                    if (completed === promises.length) {
                        resolve(results);
                    }
                });
        });
    });
}
Пояснение

allSettled не падает при первом reject. Каждый промис превращается в объект результата со статусом, а итоговый промис завершается только после обработки всех элементов.

class EventEmitter {
    constructor() {
        this.events = new Map();
    }

    on(eventName, handler) {
        if (!this.events.has(eventName)) {
            this.events.set(eventName, new Set());
        }

        this.events.get(eventName).add(handler);
    }

    emit(eventName, ...args) {
        const handlers = this.events.get(eventName);
        if (!handlers) return;

        handlers.forEach((handler) => handler(...args));
    }

    off(eventName, handler) {
        this.events.get(eventName)?.delete(handler);
    }
}
Пояснение

Map хранит события, а Set хранит обработчики без дублей. on добавляет обработчик, emit вызывает все обработчики события, off удаляет конкретную функцию.

Решение
async function runWithLimit(tasks, limit) {
    const results = [];
    let nextIndex = 0;

    async function worker() {
        while (nextIndex < tasks.length) {
            const currentIndex = nextIndex;
            nextIndex += 1;

            results[currentIndex] = await tasks[currentIndex]();
        }
    }

    const workers = Array.from(
        { length: Math.min(limit, tasks.length) },
        () => worker(),
    );

    await Promise.all(workers);
    return results;
}
